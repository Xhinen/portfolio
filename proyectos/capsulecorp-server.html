Arquitectura del entorno

El servidor concentra los servicios de la empresa detrás de un firewall, con los usuarios organizados en tres grupos por rol y una capa de automatización y seguridad por encima.

Objetivos

  • Segmentar el acceso de los usuarios por roles (administradores, desarrolladores y soporte).
  • Desplegar y configurar los servicios clave: SSH, Apache2 y MariaDB.
  • Automatizar tareas repetitivas (altas de usuarios, backups y monitorización) con scripts y cron.
  • Aplicar hardening siguiendo el principio de mínimo privilegio.

Usuarios, grupos y permisos ACL

Cada departamento tiene su propio grupo. Los usuarios se crean con su directorio home y su shell, y se asignan al grupo que les corresponde:

GrupoUsuariosRol
administradoresadmin1..4Diseño y mantenimiento de la infraestructura
desarrolladoresdev1..4Construyen y mantienen las aplicaciones
soportesupp1..4Resuelven incidencias (acceso de lectura)
bash
$ sudo groupadd administradores
$ sudo groupadd desarrolladores
$ sudo groupadd soporte

# Crear usuario con home (-m) y shell (-s), asignado a su grupo (-G)
$ sudo useradd -m -s /bin/bash -G administradores admin1

# Listar todos los grupos y sus usuarios de un vistazo
$ getent group | awk -F: '$4 != "" {print "["$1"] -> "$4}'

Permisos avanzados con ACL

El sistema solo permite un usuario y un grupo propietario. Para un escenario realista en el que dev1 y el grupo soporte también necesitan acceso al directorio compartido, se usan ACL (Access Control List):

bash
$ sudo mkdir /srv/capsulecorp
$ sudo chown root:desarrolladores /srv/capsulecorp
$ sudo chmod 770 /srv/capsulecorp

# Permisos extra para un usuario y un grupo concretos
$ sudo setfacl -m u:dev1:rwx /srv/capsulecorp
$ sudo setfacl -m g:soporte:r-x /srv/capsulecorp

# ACL por defecto: se hereda en los ficheros nuevos del directorio
$ sudo setfacl -d -m g:desarrolladores:rwx /srv/capsulecorp

Además se reforzó la gestión de cuentas bloqueando usuarios (usermod -L) y estableciendo caducidad de contraseña (chage -M 90).

Servicios del sistema

SSH securizado

El servicio SSH se endurece cambiando el puerto por defecto, deshabilitando el acceso de root, limitando los intentos y restringiendo qué usuarios pueden conectarse:

/etc/ssh/sshd_config
Port 2222
PermitRootLogin no
PasswordAuthentication yes
MaxAuthTries 3
AllowUsers admin1 dev1
Banner /etc/ssh/banner.txt

Servidor web Apache2

Se crea un VirtualHost propio para servir la web de la empresa (la tienda TechShop del módulo de aplicaciones web se aloja precisamente aquí):

/etc/apache2/sites-available/capsulecorp.conf
<VirtualHost *:80>
    ServerName capsulecorp.local
    DocumentRoot /var/www/capsulecorp
    ErrorLog ${APACHE_LOG_DIR}/capsulecorp_error.log
    CustomLog ${APACHE_LOG_DIR}/capsulecorp_access.log combined
</VirtualHost>

Base de datos MariaDB

Tras instalar MariaDB se ejecuta mysql_secure_installation para eliminar usuarios anónimos, deshabilitar el login remoto de root y borrar la base de datos de prueba. Después se crea una base de datos con su usuario y privilegios acotados.

Automatización con scripts y cron

Tres scripts en Bash cubren las tareas repetitivas del servidor, todos protegidos para que solo los administradores puedan ejecutarlos:

  • gestion_usuarios.sh — menú interactivo para crear, eliminar y listar usuarios, con validación del grupo.
  • backup_sistema.sh — empaqueta /srv, /etc/apache2 y /etc/ssh en un .tar.gz con fecha y limpia los backups de más de 7 días.
  • monitorizacion_sistema.sh — actualiza paquetes, limpia caché y temporales, y deja un informe en /var/log.

Los dos últimos se programan en cron para que se ejecuten solos:

crontab -e
# Copia de seguridad diaria a las 3:00
0 2 * * * /usr/local/sbin/backup_sistema.sh

# Mantenimiento semanal (domingos a las 23:00)
0 23 * * 7 /usr/local/sbin/mantenimiento_sistema.sh

Seguridad y hardening

La capa de seguridad combina varias técnicas: shells deshabilitadas para usuarios de servicio, permisos correctos en la web, auditoría de binarios con SUID y un firewall con política restrictiva.

UFW · firewall
$ sudo ufw default deny incoming
$ sudo ufw default allow outgoing
$ sudo ufw allow 2222/tcp comment 'SSH'
$ sudo ufw allow 80/tcp   comment 'HTTP'
$ sudo ufw allow 443/tcp  comment 'HTTPS'
$ sudo ufw enable

# Auditar binarios con bit SUID cada cierto tiempo
$ find / -perm /4000 -type f 2>/dev/null

Extra: sudo Zero Trust + Fail2ban

Como capa adicional (aprovechando la formación en ciberseguridad), se aplica una política Zero Trust sobre sudo usando ficheros modulares en /etc/sudoers.d/: se pide contraseña siempre, se graban las sesiones y cada rol solo puede ejecutar sus comandos.

/etc/sudoers.d/01-defaults-zerotrust
Defaults    env_reset
Defaults    timestamp_timeout=0      # pedir contraseña siempre
Defaults    passwd_tries=3
Defaults    requiretty
Defaults    logfile="/var/log/sudo.log"
Defaults    log_input, log_output    # grabar sesiones (auditoría)

Y Fail2ban protege el SSH baneando las IP que fallan al autenticarse:

/etc/fail2ban/jail.local
[sshd]
enable   = true
port     = 2222
maxretry = 3
bantime  = 86400
findtime = 86400
backend  = systemd

Retos y aprendizajes

💡

Tras cambiar el puerto de SSH a 2222, el servicio no aceptaba conexiones aunque lo reiniciara. La solución fue reiniciar el servidor por completo: a veces el cambio de puerto no se aplica solo con systemctl restart ssh. Un recordatorio de comprobar siempre el estado real del servicio antes de dar algo por hecho.
