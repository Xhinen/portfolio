Administración avanzada de MariaDB

● completado MariaDB SQL RBAC Triggers DBeaver

Diseño e implementación de una base de datos relacional para la empresa Capsulecorp sobre MariaDB: modelo de datos, seguridad basada en roles, copias de seguridad automatizadas, optimización de consultas y automatización con triggers, procedimientos y funciones.

Motor
MariaDB 10.11 · InnoDB
Servidor
Ubuntu Server (VM)
Cliente
DBeaver (remoto)
Tablas
6 + integridad referencial

Objetivos

  • Diseñar un modelo relacional coherente con claves foráneas e integridad referencial.
  • Aplicar seguridad basada en roles (RBAC) con el principio de mínimo privilegio.
  • Garantizar la recuperación de datos con backups automatizados.
  • Optimizar el rendimiento con índices y automatizar la lógica con triggers, procedimientos y funciones.

Instalación y configuración

Tras instalar MariaDB se ejecuta mysql_secure_installation (eliminar usuarios anónimos, bloquear root remoto, borrar la BD de prueba). La configuración del servidor se afina en su fichero principal:

/etc/mysql/mariadb.conf.d/50-server.cnf
[mysqld]
bind-address           = 127.0.0.1   # escuchar solo en local
character-set-server   = utf8mb4      # Unicode completo (4 bytes)
collation-server       = utf8mb4_spanish_ci
default-storage-engine = InnoDB
innodb_buffer_pool_size = 1G
slow_query_log         = 1            # registrar consultas lentas
long_query_time        = 2

Modelo de datos

La base de datos se compone de 6 tablas que cubren las necesidades de la empresa:

TablaDescripción
departamentosDepartamentos de la empresa
empleadosDatos de los empleados (FK a departamento)
proyectosProyectos gestionados por cada departamento
asignacionesRelación N:M entre empleados y proyectos
gastosGastos asociados a cada proyecto
auditoriaRegistro automático de cambios (triggers)

Relaciones principales: departamentos → empleados (1:N), departamentos → proyectos (1:N), empleados ↔ proyectos (N:M vía asignaciones) y proyectos → gastos (1:N). La base de datos usa utf8mb4 e InnoDB para soportar transacciones y claves foráneas.

SQL
CREATE DATABASE capsulecorp
  CHARACTER SET utf8mb4
  COLLATE utf8mb4_spanish_ci;

Seguridad basada en roles (RBAC)

En lugar de asignar permisos usuario por usuario, se definen 4 roles (rol_dba, rol_desarrollo, rol_direccion, rol_soporte) y se asignan a los usuarios. Así, añadir un empleado nuevo es trivial y se reducen los errores.

SQL · roles y usuarios
-- Definir el rol y sus permisos una sola vez
CREATE ROLE IF NOT EXISTS rol_desarrollo;
GRANT SELECT ON capsulecorp.* TO rol_desarrollo;
GRANT INSERT, UPDATE ON capsulecorp.proyectos TO rol_desarrollo;

-- Crear el usuario y asignarle el rol
CREATE USER 'carlos_dev'@'%' IDENTIFIED BY '********';
GRANT rol_desarrollo TO 'carlos_dev'@'%';

-- Clave: activar el rol por defecto (si no, parece que no funciona)
SET DEFAULT ROLE rol_desarrollo FOR 'carlos_dev'@'%';

Copias de seguridad

Las copias se realizan con mysqldump y se automatizan con un script programado en cron, que además limpia los backups con más de 30 días.

bash
# Backup completo de la base de datos
$ mysqldump -u dba_admin -p capsulecorp > backup_$FECHA.sql

# Restauración: crear la BD y volcar el .sql
$ mysql -u root -p -e "CREATE DATABASE capsulecorp;"
$ mysql -u root -p capsulecorp < backup_$FECHA.sql

# Programado en cron (cada día a las 2:00)
0 2 * * * /home/kaneda/scripts/backup_auto_mariadb.sh

Consultas avanzadas y optimización

Se crean índices en las columnas usadas en WHERE y JOIN, y se verifican los planes de ejecución con EXPLAIN. Ejemplo con JOIN, GROUP BY y HAVING (empleados en más de un proyecto):

SQL
SELECT e.nombre, e.apellidos,
       COUNT(a.id_proyecto) AS num_proyectos
FROM empleados e
INNER JOIN asignaciones a ON e.id_empleado = a.id_empleado
GROUP BY e.id_empleado
HAVING COUNT(a.id_proyecto) > 1
ORDER BY num_proyectos DESC;

Automatización: triggers y funciones

La lógica de negocio se automatiza dentro de la propia base de datos: un trigger audita los cambios de salario, otro bloquea la inserción de un gasto si supera el presupuesto del proyecto (con SIGNAL), más un procedimiento de informe y una función de porcentaje consumido.

SQL · trigger de control de presupuesto
CREATE TRIGGER trg_control_presupuesto
BEFORE INSERT ON gastos
FOR EACH ROW
BEGIN
    DECLARE gasto_actual DECIMAL(12,2);
    SELECT COALESCE(SUM(importe), 0) INTO gasto_actual
    FROM gastos WHERE id_proyecto = NEW.id_proyecto;

    IF (gasto_actual + NEW.importe) > presupuesto_proy THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'El gasto supera el presupuesto del proyecto.';
    END IF;
END

Retos y aprendizajes

💡

Los usuarios se conectaban pero "parecía" que no tenían permisos, aunque el rol estuviera concedido. La razón: MariaDB no activa los roles automáticamente. La solución fue usar SET DEFAULT ROLE para cada usuario. Un detalle pequeño que enseña a leer bien la documentación oficial.

Conclusión

Una práctica muy completa que va del diseño del modelo hasta la automatización con triggers, pasando por seguridad, backups y optimización. Me ayudó a entender por qué las bases de datos son tan estrictas con permisos e integridad, y reforzó mi base de SQL para el día a día como administrador.
